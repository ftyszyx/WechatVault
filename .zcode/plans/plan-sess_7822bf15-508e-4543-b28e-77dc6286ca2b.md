
# WeChat Data Export Demo - Implementation Plan

## Overview
Build on the existing key-finding PoC to add actual database decryption and message export functionality. The demo will find the SQLCipher key from WeChat process memory, use it to open the encrypted `message_0.db`, and export chat messages to JSON.

## Architecture

### Two approaches considered:
| Approach | Pros | Cons |
|---|---|---|
| **A) rusqlite + bundled-sqlcipher** | Clean, reliable, handles all SQLCipher details | Requires C compilation of SQLCipher |
| **B) Manual page decryption + plain rusqlite** | No SQLCipher dependency | Must reimplement page-level AES/IV/HMAC logic, fragile |

**Chosen: Approach A** — `rusqlite` with `bundled-sqlcipher` feature. This is the standard way to work with SQLCipher databases from Rust and handles all page crypto correctly.

### 2. `src/main.rs` — Add export subcommand and flow
- Add `Commands` enum with `Export` variant (clap subcommands)
- `Export` subcommand args: `--output` (path, default `messages.json`), `--table` (optional, for specifying which table to export)
- In the match-success code path, extract the actual key bytes and pass them to the export module
- Currently the key is zeroized immediately after match — need to capture it before zeroizing

### 3. New file: `src/export.rs` — Database export logic
Functions:
- `open_encrypted_db(db_path, key)` — Opens message_0.db with SQLCipher using `PRAGMA key`
- `list_tables(conn)` — Lists all tables for discovery
- `export_table_to_json(conn, table_name, output_path)` — Exports a table's rows as JSON array
- `export_messages(conn, output_path)` — Smart export: tries `message` table first, falls back to listing all tables

### 4. New file: `src/key.rs` — Key management helpers
- Move `CandidateSet` and key-related types here for cleaner separation
- `KeyStore` type that holds the found key and handles zeroize-on-drop

## Flow

```
1. Parse CLI args (now with subcommands: `verify` or `export`)
2. Locate Weixin.exe PID and message_0.db (existing logic)
3. Scan process memory for PEM markers (existing logic)
4. Collect key candidates from memory (existing logic)
5. Validate candidates against DB page HMAC (existing logic)
6. [NEW] If MATCH and export mode:
   a. Capture the raw key bytes (before zeroize)
   b. Open message_0.db via rusqlite+sqlcipher with key as PRAGMA
   c. Discover tables, export message data to JSON
   d. Zeroize key
7. Report results
```

## Key design decisions

1. **Key extraction**: Currently the verification functions (`verify_raw_candidate`, `verify_derived_candidate`) only return `MatchKind`. Need to also return the raw key bytes that matched. Modify to return `Option<(MatchKind, [u8; 32])>`.

2. **SQLCipher PRAGMA**: `PRAGMA key = "x'<hex_key>'"` with the raw key (not derived). SQLCipher handles PBKDF2 derivation internally.

3. **WeChat schema**: WeChat 4.x `message_0.db` typically has tables like `message`, `chat`, `contact`, `chatroom`, etc. The exporter will try the `message` table first and handle missing tables gracefully.

4. **Output format**: JSON array of objects, each row becomes a JSON object with column_name → value mapping. Timestamps converted to ISO 8601 via `chrono`.

5. **Safety**: Key is zeroized after export. No key is ever written to disk. The temporary decrypted data only exists in SQLCipher's in-memory pages (no temp file).
